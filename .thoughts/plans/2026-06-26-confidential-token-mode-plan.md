# Plan: Confidential Token Mode (ZK Freighter Track B / B-P)

Architecture-decision gate. The single architecture-blocking unknown is **resolved** (B1 PASSED: our `nargo 1.0.0-beta.11`+`bb 0.87.0` reproduce the on-chain VK byte-for-byte; `noir_js`+`bb.js@0.87.0` prove/verify; bb.js VK == on-chain VK). This plan decides how to build; it does not begin implementation.

## Inputs
- Spec `2026-06-26-confidential-token-mode-spec.md`; Stories `2026-06-26-confidential-token-mode-stories.md`.
- Research: `2026-06-25-confidential-token-repo-study.md`, `2026-06-26-confidential-token-deep.md`, `2026-06-26-confidential-token-b1-spike.md`.
- Overall plan (Track B); repo: `packages/core` (TS), `packages/ui`, `apps/web` redesigned shell; reference (study-only) `reference/openzeppelin-stellar-contracts-confidential` @ `539968f`.

## Assumptions
- Toolchain stays pinned (C4). Reproduced circuits/contracts carry attribution + tests; **nothing imported from `reference/`**.
- Confidential mode is fully isolated from the shielded-pool path (separate keys/prover/state) — additive, no shielded-pool changes.
- Testnet-only; no mainnet; no clawback.

## Open Questions → Resolutions (recommended)
- Q1 demo underlying → **deploy a fresh demo SEP-41-wrapped token** (clean, controlled evidence).
- Q2 recovery/indexer → **in-wallet scoped event scanner** over Soroban RPC `getEvents` window, persisting an encrypted opening store; **honest "history incomplete" fallback** when the window doesn't cover full history. No separate service for v1 (revisit if the RPC window proves too short).
- Q3 auditor identity → **demo wallet registers its own auditor key** (distinct from spend keys) so dual-auditor disclosure is demonstrable end-to-end; the auditor key cannot spend.
- Q4 key model → **Grumpkin confidential-account keys derived from the seed via a distinct domain/path**, isolated from BN254 note-keys; `@noble/curves` short-Weierstrass for Grumpkin (`y²=x³−17`), validated in B2 against reference testdata; Poseidon2-t4 for viewing-key derivation.
- Q5 mode placement → **a top-level "Confidential" sidebar destination** with its own sub-flows; Activity tags entries by mode (shielded vs confidential).
- Q6 on-chain `verify_proof` of our proof → **first funded step (Phase 3)**.

## Prototype Reintegration Gate
The prototype does NOT cover confidential mode (confirmed). There is therefore no prototype-mock to carry; screens are designed fresh in the `packages/ui` language (a flagged designer gap). **No-shipping-mock policy:** the judged path uses real contracts/proofs/balances; provenance + crypto KATs are local test fixtures only (excluded from the product path). Until contracts are deployed (funding), UI shows honest "not deployed / register first" states — never fabricated data.

## Phase 2: Key model + crypto primitives + prover wrapper (no funding)
### Goal
Seed-derived Grumpkin confidential-account key model, Poseidon2-t4, and the gadget primitives (commit / ECDH / encrypt_amount / vk_from_sk), each KAT-verified against reference `circuits/lib/testdata/*.json`; plus the browser prover wrapper.
### Work
- `packages/core/src/confidential/keys.ts`: derive `(sk, viewing key, address_as_field)` on Grumpkin from the seed via a distinct domain; **isolated** from `identity.ts` note-keys.
- `confidential/crypto.ts`: Poseidon2-t4 + Grumpkin ops (via `@noble/curves`), validated on-curve/non-identity; KATs vs `testdata/{commit,ecdh,encrypt_amount,vk_from_sk,...}.json`.
- `confidential/prover.ts`: `UltraHonkBackend` (bb.js@0.87.0) + `noir_js` wrapper, circuit bytecode + VK assets resolved per-circuit (served like `/js` for nethermind), with `onStatus` for the proof-flow ring. Isolated runtime.
### Real Integration Path
Real Grumpkin/Poseidon2 against reference test vectors; real proof generation for at least one op (e.g. `register`) with a constructed valid witness.
### Mock/Simulation Policy
KATs/fixtures are test-only. No fabricated keys/openings in product code.
### Checks
KATs pass; `@noble/curves` Grumpkin coverage confirmed (resolves research Q); TDD on key derivation; one op proves+verifies in JS.
### Acceptance Criteria Covered
AC1 (deepened), AC12 (KATs).
### Stop Condition
Grumpkin/Poseidon2/gadget KATs green; one circuit proven via the wrapper.
