reference implementation (privacy-pools / Tornado-Nova–style 2-in/2-out transfer):
prover stack, artifacts, browser proof generation path, witness vs. proof split,
worker model (Web Worker vs. MV3 service worker), proof formatting for the on-chain
BN254 verifier, and whether the circuits + keys are reusable as-is. Facts verified
from the cloned repo at `/Users/abu/dev/hackathon/stellar-research/repos/stellar-private-payments`.
External snarkjs/Tornado-Nova references included for comparison only. **This brief
documents current reality only; it proposes no solution, port, or design.**

## Verified Facts

### Prover stack — arkworks in Rust→WASM, NOT snarkjs/rapidsnark
- The proof is generated by **`ark-groth16` v0.6.0** over **`ark-bn254` v0.6.0**, compiled
  to `wasm32` and run in the browser. Source: `prover.rs:19-28,343` uses
  `ark_groth16::Groth16<Bn254, CircomReduction>::prove(...)`; versions from
  `Cargo.toml:30-33` and `Cargo.lock` (`ark-groth16 0.6.0`, `ark-bn254 0.6.0`).
- snarkjs and rapidsnark are **not** used for proving. snarkjs JSON appears only as a
  *verifying-key export format* (`circuit-keys/src/lib.rs:113-134` `write_vk_snarkjs_json` /
  `vk_to_snarkjs_json`), and `deployments/testnet/circuit_keys/policy_tx_2_2_vk.json` is
  snarkjs-shaped (`"protocol":"groth16"`, `"curve":"bn128"`, `"nPublic":11`).
- `prover.rs:5-10` documents the explicit design reason: *"We cannot use ark_circom
  directly because it depends on wasmer which doesn't work in browser WASM."* The repo
  therefore **splits the pipeline**: witness calc in one component, proving in another.

### Two-stage split: witness calculation vs. proof generation
- **Witness calculation** uses the **Circom-emitted `.wasm`** executed by **`wasmer`**, via a
  **Nethermind fork of `ark-circom`**. Source `witness/src/lib.rs:1-13,42-97`:
  `ark_circom::{WitnessCalculator, circom::R1CSFile}` + `wasmer::{Module, Store}`.
- The ark-circom dependency is a fork, not crates.io: `Cargo.toml:31`
  `ark-circom = { git = "https://github.com/NethermindEth/circom-compat.git", branch = "wasm-no-parallel" }`;
  `Cargo.lock` resolves it to `ark-circom 0.5.0` at commit `ef9b618…905fb3`.
- wasmer is also a pinned git build: `Cargo.toml:75`
  `wasmer = { git = "https://github.com/wasmerio/wasmer", rev = "763e9f28…16ddd1" }`;
  `Cargo.lock` → `wasmer 7.2.0-alpha.2`.
- **Per-target wasmer backend** (`witness/Cargo.toml:23-30`):
  - non-wasm32: `wasmer` features `["cranelift"]` + ark-circom `["parallel"]`.
  - **wasm32 (browser): `wasmer` features `["js","js-default"]`** + ark-circom default (no parallel).
  So in-browser, the Circom witness `.wasm` runs **on the JS engine's own WebAssembly**
  via wasmer's `js` backend (wasmer is the host shim around `WebAssembly`), while the
  surrounding Rust is the app's own wasm module. Comment `witness/Cargo.toml:25` cites
  upstream issue #192.
- **Proof generation** does NOT run the Circom wasm. It parses the **`.r1cs`** file into
  constraint matrices and **replays** them as an arkworks `ConstraintSynthesizer`
  (`prover.rs:117-231`, struct `R1CSCircuit`), feeding the pre-computed witness values,
  then calls `ark-groth16::prove`. R1CS parsing is custom (`app/crates/core/prover/src/r1cs.rs`).

### Artifacts required at proving time (per circuit)
For each circuit the prover worker needs **three** artifacts:
1. **Circuit `.wasm`** (Circom witness generator) — fetched at runtime over HTTP.
2. **`.r1cs`** (constraint system) — fetched at runtime over HTTP; used by BOTH the
   witness calculator (header/wire counts) and the prover (constraint replay).
3. **Proving key `.bin`** — **arkworks-compressed Groth16 proving key**
   (`ProvingKey::<Bn254>::serialize_compressed`, `circuit-keys/src/lib.rs:105-111`),
   **embedded in the worker wasm at compile time** via `include_bytes!`
   (`prover.rs:24-29`). It is NOT the snarkjs `.zkey` format.
- Concrete sizes (`deployments/testnet/circuit_keys/`, `ls -la`):
  - `policy_tx_2_2_proving_key.bin` = **8,126,128 bytes (~8.1 MB)** (the 2-in/2-out tx circuit).
  - `selectiveDisclosure_1_proving_key.bin` = **1,129,680 bytes (~1.1 MB)**.
- `.wasm` and `.r1cs` are **not committed**; they are built by `cargo build -p circuits`
  into `target/circuits-artifacts/<profile>/` (`build.rs`, `Trunk.toml` pre_build hook),
  then copied to `dist/circuits/` and fetched at runtime from `circuits/policy_tx_2_2.wasm`
  / `.r1cs` (`prover.rs:89-120`, `Trunk.toml` hook). Exact built `.wasm`/`.r1cs` sizes are
  therefore **not present in the cloned repo** (see Unknowns).
- **Integrity pinning:** the worker SHA-256-checks every artifact (proving key, wasm, r1cs)
  against compile-time constants before use (`prover.rs:48-162`; constants generated by
  `web/build.rs` into `artifact_hashes.rs`). A version mismatch hard-fails proving.

### How a proof is generated in the browser (call path)
- JS only calls `init()` then `mainThread(config)` (`app/js/wasm-facade.js:1-13`). JS
  **never** talks to workers directly (`ARCHITECTURE.md:26`).
- The Rust **main-thread WASM module spawns the workers and routes messages**; the worker
  protocol is intentionally not exposed to JS (`ARCHITECTURE.md:42,56-58`).
- A transfer/transact proof is requested via `ProverWorkerRequest::Transact(TransactParams)`
  (`protocol.rs:107-109`). In the worker (`prover.rs:236-241,342-398`):
  1. `transact(params, hash_ext_data_offchain)` builds circuit inputs + ext data
     (`prover::flows::transact`).
  2. inputs serialized to JSON → `WitnessCalculator::compute_witness(json)` →
     **witness bytes (32-byte little-endian field elements)** (`witness/src/lib.rs:74-97,317-354`;
     `prover.rs:346-353`).
  3. `Prover::prove_bytes(witness)` → compressed proof; then `extract_public_inputs`,
     `verify` (self-check), then `proof_bytes_to_uncompressed` (`prover.rs:362-377`).
  4. returns `PreparedProverTx { proof_uncompressed (256 bytes), ext_data, prepared, soroban_tx }`
     (`protocol.rs:180-193`).
- Artifacts are loaded once and cached in `thread_local!` `RefCell<Option<…>>` slots
  (`prover.rs:76-189`); `worker_main` eagerly calls `load_circuit_artifacts()` on startup
  and a `Ping` busy-waits until ready (`prover.rs:192-235`).

### Worker model — gloo Web Worker, NOT an MV3 service worker
- Proving runs in a **dedicated Web Worker** built with **`gloo_worker`** in oneshot mode
  (`prover.rs:7` `use gloo_worker::{Registrable, oneshot::oneshot}`; `#[oneshot] ProverWorker`
  at `prover.rs:211-217`; `ProverWorker::registrar().register()` at `prover.rs:196`).
- Workers are declared/built by **Trunk** as separate wasm bins (`app/index.html:34-35`):
  `data-trunk rel="rust" … data-bin="prover-worker" data-type="worker" data-bindgen-target="web"`.
  A separate `storage-worker` exists too.
- Inside the worker, async work runs on `wasm_bindgen_futures::spawn_local` and HTTP fetches
  use `web_sys::WorkerGlobalScope::fetch_with_request` (`prover.rs:17,197,431-434`) — i.e.
  the worker uses the **DedicatedWorkerGlobalScope** fetch, not a page `window`.
- `ARCHITECTURE.md:51-54`: the Prover Worker exists specifically to run "long/blocking
  proving so it cannot block" other work; "loads/caches proving artifacts in memory."
- **No MV3 / browser-extension service worker is used anywhere.** Repo-wide grep for
  `service_worker` / `navigator.serviceWorker` / `serviceWorker` returned **zero matches**
  in `app/js`, `app/*.html`, and `app/crates/platforms/web/src`. This is a standard web
  app (Trunk + Next-less static HTML + esbuild JS bundles per `Trunk.toml`), not a browser
  extension.

### On-chain verifier + proof formatting (BN254, Soroban)
- The verifier is a **Soroban smart contract** using the chain's **native BN254 precompile**
  (`contracts/circom-groth16-verifier/src/lib.rs:18-23,99-140`): `env.crypto().bn254()`,
  `g1_mul`/`g1_add`, and `pairing_check`. Pairing check is the standard Groth16 equation
  `e(-A,B)·e(alpha,beta)·e(vk_x,gamma)·e(C,delta) == 1` (`lib.rs:122-135`).
- The VK is **embedded in the verifier contract wasm at compile time** from a snarkjs
  `verification_key.json` via `build.rs` + `VERIFIER_VK_JSON` env (`lib.rs:6-13,25-26`).
- **Proof wire format expected on-chain = uncompressed BN254, 256 bytes total:**
  `A (G1, 64 bytes) || B (G2, 128 bytes) || C (G1, 64 bytes)`
  (`prover.rs:74-83,382-424`). G1 = `x(32 BE)||y(32 BE)`; G2 uses **Soroban/Ethereum
  ordering `c1||c0` (imaginary||real)** per coordinate: `x.c1||x.c0||y.c1||y.c0`
  (`prover.rs:56-72` `g2_bytes_uncompressed`). The worker asserts the result is exactly
  256 bytes (`prover.rs:371-376`) and returns it as `proof_uncompressed`.
- **Public inputs:** the 2-in/2-out circuit `policy_tx_2_2` has **11 public signals**
  (`policy_tx_2_2_vk.json` `"nPublic":11`). The public set is declared in
  `policy_tx_2_2.circom`: `root, publicAmount, extDataHash, inputNullifier[2],
  outputCommitment[2], membershipRoots[2×1], nonMembershipRoots[2×1]`. The worker forwards
  these as `PreparedTxPublic { pool_root, input_nullifiers[2], output_commitments[2],
  public_amount, ext_data_hash_be, asp_membership_root, asp_non_membership_root }`
  (`protocol.rs:168-178`).
- The VK is also emitted in two extra formats for on-chain use: a Soroban binary layout
  (`circuit-keys/src/lib.rs:178-211`, alpha/beta/gamma/delta + IC list) and a Rust-const
  source file (`:214-252`).

### The specific circuit (2-in / 2-out transfer)
- Entry point `policy_tx_2_2.circom`: `PolicyTransaction(2, 2, 1, 1, 10, 10)` — **2 inputs,
  2 outputs**, 1 membership proof + 1 non-membership proof per input, Merkle depth 10,
  SMT depth 10. Pragma `circom 2.2.2`.
- It is a **Tornado-Nova-style UTXO transfer circuit extended with ASP membership /
  non-membership** (`policyTransaction.circom`): Poseidon2 commitments, keypair/signature,
  Merkle proof per input, nullifier uniqueness, `sumIns + publicAmount === sumOuts`,
  248-bit output range check. Hash = **Poseidon2** (custom `poseidon2/` circom + a
  `poseidon2` crate), not the classic Poseidon used in stock Tornado-Nova.

### Reusability of the Nethermind circuits + keys (verified properties)
- **Circuits (.circom):** Apache-2.0 / present in-repo; LGPL notices are bundled for the
  circomlib-derived parts (`deployments/legal/dist/circuits-NOTICE.txt`, `circomlib.lock`).
  They are standard `circom 2.2.2` source and compile to standard `.wasm`/`.r1cs`.
- **`policy_tx_2_2` keys (the 2-in/2-out tx keys):** produced by a **trusted ceremony**
  (`circuit-keys/.../README.md` → ceremony issue #177). The proving key is **arkworks-
  compressed bin**, and a matching **snarkjs VK JSON + Soroban VK** are provided. The VK is
  baked into the on-chain verifier — *"Changing the `policy_tx_2_2` keys requires redeploying
  the on-chain verifier and any dependent contracts."*
- **`selectiveDisclosure_1` keys:** **locally generated, NOT from a trusted ceremony**
  (README "Provenance caveat") — "suitable for testnet/off-chain disclosure receipts only."
- The proving key, witness `.wasm`, and `.r1cs` are **version-locked together** by SHA-256
  pins in `web/build.rs`; any mismatch fails at runtime (`prover.rs:48-162`). Reuse requires
  carrying the matched artifact triple + the corresponding VK in the verifier.
- Curve is **BN254/bn128** throughout (circuits, keys, verifier) — `policy_tx_2_2_vk.json`
  `"curve":"bn128"`; verifier uses `crypto::bn254`.

### Comparison data point (external, snarkjs path — NOT this repo)
- A comparable snarkjs-based 2-in/2-out privacy circuit (Sable, Tornado-Nova-derived)
  reports browser Groth16 proving **~10–15 s**, with `transaction.wasm` ~2.1 MB and a
  snarkjs `circuit_final.zkey` ~14 MB. Source: Sable proving/verification docs
  (https://mintlify.wiki/collinsville22/sablev2/circuit/proving-verification). **This is a
  different stack (snarkjs, not arkworks/wasmer) and is provided only as an order-of-magnitude
  reference, not a measurement of the Nethermind implementation.**

## Inferences

- **(Inference, medium confidence)** Because the browser path runs the Circom witness `.wasm`
  through wasmer's `js` backend and the Groth16 prover through `ark-groth16` with **no
  parallelism** (ark-circom branch `wasm-no-parallel`, wasm32 target has no `parallel`/rayon
  feature; `witness/Cargo.toml:28-30`, `Cargo.toml:31`), proving is **single-threaded** in the
  worker. This typically makes WASM proving slower than native and slower than multi-threaded
  snarkjs builds. No in-repo benchmark confirms a number.
- **(Inference, medium confidence)** Peak memory is dominated by the **~8.1 MB compressed
  proving key** decompressed into arkworks structures (`Prover::new` uses
  `deserialize_compressed_unchecked`, `prover.rs:252-273`) plus the full witness vector
  (one `Fr` per wire) and constraint matrices held in RAM. Order-of-magnitude this is tens
  to low-hundreds of MB for a circuit of this size, but the repo states no figure.
- **(Inference, high confidence)** Running this exact pipeline inside an **MV3 extension
  service worker would be constrained** by MV3's service-worker lifecycle (idle termination,
  historically ~30 s, and no DOM). The repo's design already isolates proving in a *dedicated
  Web Worker* (not a service worker) precisely because proving is "long/blocking"
  (`ARCHITECTURE.md:51-54`). A long single-threaded proof in an MV3 SW risks the SW being
  killed mid-proof. This is an inference about MV3 generally; the repo does not target MV3.
- **(Inference, high confidence)** The circuits + `policy_tx_2_2` keys are **reusable as-is on
  any BN254 Groth16 verifier** (the VK JSON is snarkjs-standard), but reuse **off Soroban**
  would need re-formatting proof/VK byte order (the repo hard-codes Soroban's `c1||c0` G2
  ordering, `prover.rs:59-72`).
