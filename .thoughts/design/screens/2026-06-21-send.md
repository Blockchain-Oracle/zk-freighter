# Send (private) — screen cluster spec

> Product: **ZK Fighter** (brand locked). Privacy-by-default self-custody ZK wallet for **shielded** payments on Stellar. Web app first.
> Cluster owner: this spec covers the **private send** flow only — moving value PRIVATE → PRIVATE (shielded balance → a recipient's private address). Shield (public→private) and Unshield (private→public) are separate clusters; this flow never touches a public Stellar address as the destination.
> Visual direction (color/type/motion) is intentionally LEFT FREE. This spec fixes IA, components, copy, states, and interactions only.

## What this flow IS and ISN'T (so we never contradict the model)
- A private send spends from the user's **Spendable** shielded balance and lands in the recipient's shielded balance. On-chain, only a "spent" nullifier + an opaque new note commitment appear — **counterparty, amount, and memo are hidden**. This is the one genuinely-private transfer in the product, and every screen here must TEACH that.
- It requires a **recipient private address** (Railgun 0zk-style bundle = note key BN254 + encryption key X25519, shown as one string). You cannot privately send to a bare Stellar `G…` address — that would be an Unshield, a different flow. If the user pastes a `G…`, we detect it and offer to switch flows (see Recipient errors).
- A ZK proof is generated **client-side** and can take **seconds to minutes**. The flow has a first-class **Pending-proof** state; the user can navigate away while it runs. Never a frozen spinner.
- **No relayer in v1** → the account that submits the shielded tx is visible as metadata. We disclose this honestly on Review (the *fact of a transaction* and the submitting account are public; the *contents* are not).
- Assets are fixed: **XLM** and **USDC**. No fee-market UI (Stellar fees are flat/tiny). Memo is a minor, optional, **encrypted** field — never a headline.

## Flow / step machine
Borrowed 1:1 from Freighter `views/Send/index.tsx` (`STEPS` enum + in-place animated slider, visited steps persist, back/forward animation directions). Our step set:

`RECIPIENT → AMOUNT → (REVIEW) → SIGN → PENDING_PROOF → SUCCESS`
with `ERROR` reachable from REVIEW/SIGN/PENDING_PROOF, and an optional `MEMO` sub-sheet hung off AMOUNT/REVIEW.

Entry points: home "Send" action (private balance is the default context), an asset row's Send, or a tapped address-book contact (pre-fills RECIPIENT). The whole flow is a routed sub-view with a persistent header showing a **step indicator** ("Step 2 of 4"), a **back chevron**, a **close (X)**, and the **network pill (TESTNET/MAINNET)** — privacy send is testnet-only at launch, so on MAINNET the entry action is disabled-with-reason (see Disabled states).

Global rule: every numeric field uses the **Pending vs Spendable** split as its source of truth. "Max" and validation are always against **Spendable**, never total. Mark every mocked surface with a small "Mocked" tag in the prototype.

## Screen 1 — RECIPIENT ("Send privately to")
**Purpose:** capture and validate a recipient *private address*; offer paste, scan, and address book; teach that the recipient must be a registered private address (not a `G…`).
**Borrow from:** `components/send/SendTo/index.tsx` — single `Input` with left icon, 400ms debounced validation, "Suggestions / Recents / My Accounts" subheadings with identicon rows, footer Continue gated on validity. Swap Freighter's `StrKey` Stellar validation for our **private-address (Bech32m-style 0zk bundle) parser**.

**Layout & components**
- Header: title "Send privately", step "1 of 4", back = close flow.
- A one-line privacy primer chip under the header: "Recipient, amount, and memo stay hidden." (teaches the model up front; dismissible-per-session, remembered).
- **Recipient input**: full-width, left icon = shielded/lock glyph, placeholder "Paste a private address". Right adornments: **Paste** button (reads clipboard) and **Scan** (camera → QR scanner sheet). Monospace, wraps to 2 lines for the long string; shows middle-truncation `0zk1qy…h7aoo` once validated with a "valid private address" check affordance.
- **Address book / Recents** (replaces Freighter's "My Accounts"): sections "Saved contacts" and "Recent recipients" as identicon + label + truncated-address rows; tap = select + advance. A "Save to contacts" inline affordance appears after a valid never-seen address resolves (name field, optional).
- **Self-send shortcut**: a "Send to my other addresses" affordance is intentionally minor (private→private to self = consolidation; allowed but de-emphasized).
- Footer: **Continue** (secondary, rounded, full-width) — visible only when address is valid and search settled (mirror SendTo footer gating).

**States**
- *Empty:* input blank; show primer chip + Saved contacts/Recents (or their own empty states). Continue hidden.
- *Empty — no contacts/recents:* friendly line "No saved recipients yet. Paste or scan a private address to begin." (borrow Freighter "No transactions to show" tone.)
- *Loading (validating):* debounced 400ms; inline spinner in the input's right adornment; Continue hidden while unsettled (Freighter `isSearchSettled`).
- *Valid:* green check, truncated address echoed, optional contact name, "Save to contacts" affordance, Continue enabled.
- *Disabled:* Continue disabled until valid+settled. On MAINNET the whole flow is gated (see Disabled, global).
- *Error — invalid format:* warning notification "Not a private address. Private addresses start with `0zk` and are longer than a Stellar address." (borrow `InvalidAddressWarning`.)
- *Error — looks like a public `G…`/`C…` Stellar address:* SPECIAL teach-moment notification "That's a public Stellar address. Sending here would be an **Unshield** (public withdrawal) and would reveal the amount and destination. Switch to Unshield?" with a **Switch to Unshield** button (routes to that flow, carries the pasted address). This is the highest-value error in the cluster — it prevents the documented two-address footgun.
- *Error — recipient not registered:* if the address parses but the recipient has never set up a private account / published keys (mocked check), show "This private address isn't active yet. They need to open ZK Fighter once to receive privately." with a **Copy a shield-to link instead** secondary path (lets the payer use the public→private path). Continue disabled.
- *Error — clipboard/camera permission denied:* inline toast "Couldn't read clipboard / camera blocked. Paste manually." Scan sheet shows a permission-request empty state with a retry.

**Interactions**
- Paste auto-trims whitespace; Scan opens a QR sheet (camera preview, torch toggle, "Paste instead" fallback). On decode, fill + validate. Reject SEP-0007 `web+stellar:` payloads here with the "that's a public address" teach error.
- Selecting a contact/recent advances straight to AMOUNT (Freighter behavior).
- Back/X = confirm-discard only if anything entered.

**Mock data**
- Contacts: "Mom — 0zk1qy…k29f", "Freelance client — 0zk1qa…7p2x". Recents: 2 truncated `0zk…` rows. A registered-vs-unregistered toggle in the prototype to exercise the not-registered error.

## Screen 2 — AMOUNT ("How much, privately")
**Purpose:** pick asset (XLM/USDC), enter amount with **Max**, and explicitly show **what stays private**; validate against **Spendable** (not total).
**Borrow from:** `components/send/SendAmount/index.tsx` — big centered amount input with auto font-scaling, "available" balance line, asset display, Fee row, Memo affordance in the settings row, footer "Review Send" CTA that flips to "Enter an amount" when empty.

**Layout & components**
- Header: title "Amount", step "2 of 4", back → RECIPIENT.
- **Recipient summary chip** at top (identicon + name/truncated `0zk…`), tap = edit recipient.
- **Asset picker**: segmented/toggle or dropdown limited to **XLM** and **USDC** only (NO add-token, per DO-NOT-COPY). Each shows the asset's **Spendable** shielded balance. USDC trustline is handled invisibly — the word never appears; if the *recipient* lacks a USDC trustline we resolve it silently in the background and only surface a problem if it truly can't be set up.
- **Amount field**: large, auto-scaling font (Freighter `getAmountFontSize`). Below it a secondary fiat estimate line (mocked rate, tagged "est.").
- **Balance line**: "X.XX USDC spendable" (Freighter `availableBalanceText`) + a **Max** button that fills exactly Spendable. A second, dimmer line shows "+ Y.YY pending (not yet spendable)" when pending > 0 — teaches the split and explains why Max < total.
- **"What stays private" panel** (novel, required): three concise rows with hidden/visible glyphs —
  - Amount — Hidden
  - Recipient — Hidden
  - Memo — Hidden (encrypted), shown only if a memo is added
  - …and one honest counterweight row: "A transaction happened, and the submitting account, are public." (No relayer in v1.) This panel is the screen's privacy lesson; keep it lightweight, not a wall of cards.
- **Memo affordance**: a single "Add encrypted memo" text button in the settings row (Freighter Memo slot). Opens the MEMO sub-sheet. Minor by design.
- **Fee row**: "Network fee: <0.001 XLM" static; no editor (DO-NOT-COPY custom gas). If the spend would dip XLM below the Stellar base reserve, show a small reserve note.
- Footer CTA: **Review** — disabled/labelled "Enter an amount" when zero/empty (Freighter pattern).

**States**
- *Empty:* amount 0, Max available, Review labelled "Enter an amount" (disabled).
- *Loading (balance fetch):* skeleton on balance line + asset picker; Max disabled until Spendable known.
- *Valid:* amount ≤ Spendable, "What stays private" populated, Review enabled.
- *Max applied:* field = Spendable; helper "Sending your full spendable balance."
- *Disabled:* Review disabled while amount invalid/zero or balance still loading.
- *Error — insufficient spendable:* inline "More than your spendable balance. You have X.XX USDC spendable" + if pending exists, "Y.YY is still becoming spendable — try again once it clears." Offer **Use Max**. Do NOT let them spend pending funds (Zashi "shield/clear before spend").
- *Error — below dust / precision:* "Enter at least <min>." (>7 dp guard like Freighter's decimal check.)
- *Error — balance fetch failed:* notification "Couldn't load your balance" + Retry (Freighter `Failed to load send data`).
- *Pending-aware empty edge:* if Spendable = 0 but Pending > 0: "Nothing spendable yet — Y.YY is still confirming. Hang tight." Review disabled.

**Interactions**
- Asset toggle re-validates amount against the new asset's Spendable and resets Max.
- Max writes exact Spendable; editing after Max clears the "full balance" helper.
- Amount font scales down as digits grow (Freighter).
- Tapping recipient chip returns to RECIPIENT preserving state.

**Mock data**
- Spendable: 142.50 USDC, 38.20 XLM. Pending: 25.00 USDC ("clears in ~2 min"). Fiat rate mocked. A "set spendable to 0 / pending only" prototype switch to exercise edge states.

## Screen 2a — MEMO sub-sheet ("Encrypted memo") — optional
**Purpose:** attach a short note that is **encrypted to the recipient** and never public. Minor surface.
**Borrow from:** Freighter Memo editing context (`MemoEditingContext.SendPayment`) — a focused sub-sheet, not a full step.

**Layout:** sheet titled "Encrypted memo (optional)"; multiline input with char counter; helper "Only the recipient can read this. It's never posted publicly." Buttons: Save / Remove / Cancel.
**States:** empty (placeholder "Add a note for the recipient"); filled (counter, Save enabled); over-limit (counter red, Save disabled); saved (memo chip appears on AMOUNT + REVIEW as "Memo · encrypted", contents hidden behind a tap-to-view). 
**Interactions:** Save returns to caller with a memo chip; the "What stays private" panel gains the Memo=Hidden row.

## Screen 3 — REVIEW ("Confirm private send")
**Purpose:** the honesty screen — a clear, scannable **hidden-vs-public** breakdown before signing; final amounts and recipient. This is where we earn trust.
**Borrow from:** `InternalTransaction/SubmitTransaction` (TransactionConfirm) for the review layout + `goBack` to AMOUNT; Railgun/Zashi review framing for the privacy callouts.

**Layout & components**
- Header: title "Review", step "3 of 4", back → AMOUNT.
- **Summary block**: From "Your shielded balance" → To recipient (identicon + name/truncated `0zk…`); big amount + asset; fiat est.; memo chip if present.
- **Two-column "Hidden vs Public" disclosure** (the centerpiece, novel):
  - HIDDEN (with shield glyph): Amount · Recipient · Your shielded balance · Memo (if any). One line: "These never appear on-chain."
  - PUBLIC (with eye glyph, honest): "That a transaction occurred" · "The account that submits it (no relayer yet)" · "Network fee." One line: "This is the unavoidable metadata."
- **Proof notice**: "Signing starts a zero-knowledge proof on your device. It can take a few seconds to a couple of minutes — you can keep using the app while it runs." Sets expectation for PENDING_PROOF.
- **Fee line**: flat network fee, no editor.
- Footer CTA: **Confirm & sign** (primary). Secondary: Edit (→ AMOUNT).

**States**
- *Default:* all populated, CTA enabled.
- *Loading (final build/simulate):* CTA shows "Preparing…" with disabled state while the unsigned tx + proof inputs are assembled (mocked).
- *Disabled:* CTA disabled during prepare or if a late re-check (balance/recipient) invalidates.
- *Error — recipient went unregistered / stale:* "This recipient is no longer reachable privately. Go back and re-check." → Edit.
- *Error — spendable changed (race):* "Your spendable balance changed since you entered the amount. Review again." → AMOUNT with new max.
- *Error — simulate/prepare failed:* notification "Couldn't prepare this transaction" + Retry (Freighter "Failed to simulate transaction").

**Interactions:** Confirm & sign → SIGN auth. Edit/back preserves all entered data (visited-steps persistence).

## Screen 4 — SIGN (auth gate)
**Purpose:** authorize key access to start proof generation + signing. Password (default) or Face ID/passkey (if enrolled).
**Borrow from:** `views/UnlockAccount` (shared EnterPassword) for the password gate; passkey path has no Freighter reference — invent minimal WebAuthn prompt.

**Layout & components**
- Modal/sheet over REVIEW (keeps context). Title "Confirm it's you". 
- Password field (masked, show/hide) + primary **Unlock & send**; OR, if passkey enrolled, a prominent **Use Face ID** button with password as the fallback link. (Seed phrase is never requested here.)
- Small reassurance: "Your key signs locally. It never leaves this device."

**States**
- *Default:* awaiting password / Face ID.
- *Loading:* "Verifying…" after submit.
- *Error — wrong password:* inline "Incorrect password" (shake/clear), attempts not lockout-aggressive.
- *Error — passkey cancelled/failed:* "Face ID cancelled. Try again or use your password." with password fallback revealed.
- *Disabled:* Unlock disabled while password empty.
- *Success:* sheet dismisses → immediately to PENDING_PROOF.

**Interactions:** correct auth unlocks the in-memory key, kicks off client-side proving, advances to PENDING_PROOF. Cancel returns to REVIEW unchanged.

## Screen 5 — PENDING_PROOF ("Generating proof") — FIRST-CLASS STATE
**Purpose:** absorb seconds-to-minutes of client-side proving honestly; let the user **navigate away** without losing the send; never a frozen spinner.
**Borrow from:** no mainstream reference — invent from Zashi tappable status banner + Railgun proof-validity-window framing (see references doc rows: pending-proof / proving-latency).

**Layout & components**
- A determinate-where-possible **progress representation** (staged, not a bare spinner): stages "Building inputs → Generating proof → Submitting". Each stage ticks; the proof stage shows an **elapsed timer + a soft estimate** ("usually 10–90s on this device"), never a fake countdown.
- Plain-language line: "Your device is doing the private math. This is what keeps the amount and recipient hidden." (teaches WHY it's slow = a feature.)
- **"You can leave this screen" affordance**: a "Run in background" / Minimize button that drops a persistent **status pill** on Home ("1 private send proving…") which deep-links back here. The send continues if the user navigates away (within the same session/tab; if the tab is closed mid-proof, state is restored to "resume/retry" — disclose honestly).
- Cancel option (with confirm) — only valid before submission; after submit it's irreversible-pending.

**States**
- *Proving (default):* staged progress + elapsed timer + estimate; Minimize available; Cancel (pre-submit) available.
- *Slow / long-tail:* past the estimate, swap copy to "Taking longer than usual on this device — still working, safe to leave it running." No error styling.
- *Submitting:* final stage; "Broadcasting the shielded transaction." Cancel now disabled (irreversible).
- *Backgrounded:* this screen replaced by Home + a status pill; returning re-renders live progress.
- *Error — proof failed:* see ERROR screen (retry without re-entering data).
- *Success:* → SUCCESS.
- *Interrupted (tab closed mid-proof, restored):* resume banner "A private send was interrupted. Resume / Discard." Pre-submit only.

**Interactions:** Minimize → Home pill; tap pill → back here. Cancel pre-submit aborts cleanly (funds untouched, returns to Home).

**Mock data:** scripted timeline (e.g., inputs 1s → proof 12s → submit 2s) with a prototype toggle to force the slow path and the failure path.

## Screen 6 — SUCCESS ("Sent privately")
**Purpose:** confirm the private transfer, reinforce the privacy outcome, set expectations for the recipient's spendability lag.
**Borrow from:** `views/FullscreenSuccessMessage` for the success scaffold.

**Layout & components**
- Success mark + "Sent privately". Amount + asset + recipient (truncated `0zk…`/name).
- **Privacy reassurance line**: "On-chain, this looks like an opaque shielded transaction — no amount, no recipient." 
- **Recipient-spendability note** (honest, from Railgun private-transfer guidance): "Your recipient's funds become spendable shortly after their wallet syncs."
- A **balance delta** preview: "Spendable: 142.50 → 117.50 USDC" so the user sees the effect immediately; the sent amount may briefly appear as *their* pending if change notes are reorganizing — disclose if so.
- Actions: **Done** (→ Home), **View in Activity** (→ the new private row, marked pending until confirmed), **Send again** (→ fresh RECIPIENT).
- No explorer link to the *contents* (there's nothing public to show); optionally a link to the public *submission* tx with a label "This only shows that a transaction occurred."

**States**
- *Success (default):* as above.
- *Confirmed vs pending:* if the network hasn't confirmed yet, show "Submitted — confirming" with the Activity row marked pending; flip to "Confirmed" when done (Activity owns long-term status).
- *Partial/edge — change reorg:* small note if user's own spendable temporarily shows as pending.

**Interactions:** Done resets the flow (Freighter `resetSubmission`). View in Activity routes to the private row.

## ERROR screen / states (cross-cutting)
**Purpose:** recover gracefully from proof failure and pre-flight failures **without making the user re-enter the recipient/amount/memo**.
**Borrow from:** Freighter `Notification variant="error"` + `Loading`/retry scaffolding (references doc explicitly says invent failure states, borrow only generic scaffolding).

Enumerated errors and their recovery:
- **Proof generation failed** (PENDING_PROOF): "Couldn't generate the proof on this device." Causes shown plainly (low memory / interrupted). **Retry** re-runs proving with the SAME inputs (no re-entry). Secondary: "Get help". Funds never moved.
- **Submission/broadcast failed** (post-proof): "The proof was made but couldn't be submitted." Retry submission only (don't re-prove). Show whether the submitting account needs XLM for fees (and offer to fund/Shield XLM if it's empty).
- **Recipient not registered** (caught at RECIPIENT or REVIEW): see Screen 1 — offer the public→private shield-to link path.
- **Insufficient spendable** (AMOUNT/REVIEW race): see Screen 2 — never spend pending.
- **Network / RPC down:** "Can't reach the network right now." Retry; preserve all state.
- **Mainnet-gated:** if user somehow reaches the flow on MAINNET, hard stop "Private sends are testnet-only for now." with a Switch-to-testnet action; plain/unshield transfer stays available.

All error states: keep the step indicator, keep a **back-to-edit** path that preserves data, and never silently drop the in-progress send.

## Disabled / global states
- **MAINNET network selected:** Send-private entry action disabled with tooltip/inline reason "Private sends are testnet-only for now" (references doc: gate mainnet privacy with reason; plain transfer stays). Network pill always visible in the flow header.
- **Spendable = 0 (only pending or empty):** entry still allowed but AMOUNT blocks with the pending-only empty state; suggest waiting or Shielding funds.
- **No private account yet (brand-new user):** entry routes to a "Set up your private address first" nudge (one-time), not a dead end.
- **Offline:** entry disabled with "You're offline."

## Mocked-surface inventory (label each in prototype)
Recipient registration check; balance/Spendable/Pending numbers; fiat rates; proof timeline + failure injection; submission/broadcast result; recipient sync-spendability timing; activity row creation. All tagged "Mocked".

## Anti-slop checklist for this cluster
- Every step teaches the privacy model (primer chip → "What stays private" → Hidden/Public review → proof "why it's slow" → success reassurance). 
- No wall-of-identical-cards: the Hidden/Public block is a single two-column disclosure, not N cards.
- Pending-proof is a real staged state with leave-and-resume, not a spinner.
- The `G…`-pasted teach-error and the not-registered shield-link path are the two product-specific moments that prove this isn't a generic send form.
- No prescribed colors/fonts; no decorative-only motion (animations are the Freighter step-slide directions only).
