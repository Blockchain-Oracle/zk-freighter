# Activity / History — screen cluster spec

> Product: **ZK Freighter** (brand locked). Privacy-by-default, self-custody ZK wallet for shielded payments on Stellar. Web app first.
> Cluster owner: Activity / History. Designer must not prescribe colors/fonts — this spec is IA, components, states, copy, interactions only.
> All data is **MOCKED** — every surface here renders from a fixture file; mark mocked badges where noted.

## What makes THIS activity feed different (read first)

A normal wallet history is a ledger of public transfers. Ours is a **mixed-visibility log** where the wallet itself is often the only party that can read a row. Four event types, each with a different "what's public vs what's hidden" contract, and a long-running async lifecycle (client-side proving + Circle attestation) that means rows live in **pending states for seconds-to-minutes**, not just "confirming". Every row must teach, at a glance, *who else can see this*.

The four event types:

| Type | Direction | What is PUBLIC on-chain | What is HIDDEN |
|---|---|---|---|
| **Shield** (deposit) | public → private | Source Stellar account, amount, asset, the shield deposit itself | Nothing extra — this leg is inherently public |
| **Private send** | private → private | Only that *a* shielded tx happened + a "spent" nullifier marker | Sender, recipient, amount, asset, memo — all hidden. Only YOUR wallet decrypts your own copy |
| **Unshield** (withdraw) | private → public | Destination Stellar account, amount, asset, a "spent" marker, the submitting account (no relayer today → metadata leak) | Which deposit/note it came from |
| **Bridge** (add funds / on-ramp) | Ethereum → public Stellar (then shield) | Both legs are public: ETH burn tx, Circle attestation, Stellar mint. Then a separate Shield row | The bridge is NOT a private transfer — Circle is the attester |

Two visibility classes drive a per-row glyph + the detail screen's disclosure block:
- **Hidden row** (private send, and the private side of a shield/unshield): the detail screen shows *your decrypted view* and a "Hidden from everyone else" panel — there is NO explorer link for the private leg.
- **Public row / public legs** (shield source, unshield destination, bridge legs, the submitting account on unshield): explorer link(s) shown, with an honest "this is public" label.

## Screen 1 — Activity list (month-sectioned)

**Purpose.** The wallet's single chronological log of all four event types, grouped by month, each row carrying type, amount (with privacy-aware sign), status, and a visibility glyph. The default landing tab for "what has happened to my money + who could see it." Reachable from the bottom nav / home tab bar; also linked from a fresh shield/send/unshield/bridge confirmation ("View in Activity").

**Borrow from.** `reference/freighter/extension/src/popup/views/AccountHistory/index.tsx` (month-sectioned map → `AccountHistory__section-header` per `monthYear` → list of `HistoryItem`; `View.AppHeader pageTitle="History"`; empty branch `"No transactions to show"`). Row structure from `components/accountHistory/HistoryItem/index.tsx` (`__row` with left icon + `rowText` + sub-line action + right `amount` credit/debit + `date`). Pending-section + expiry-mark patterns from the Zcash UX checklist; "Incomplete" pending rows + unified shield/send/unshield tab from Railway (refs file rows "Activity — private vs public rows", "Pending / expiry in Activity").

### Layout & components
- **App header.** Back/up affordance, title "Activity". Right slot: a **Filter** button (funnel icon → opens Screen 2 filter sheet) and the persistent **network pill** (TESTNET / MAINNET) mirrored from global chrome — Activity is network-scoped, so the pill reminds the user which ledger they're reading.
- **Pending tray (conditional, pinned above month sections).** When any row is in a non-terminal state (proving, attesting, awaiting-spendable, bridge-in-flight), show a distinct **"In progress" group** at the very top — NOT inside a month. Header label "In progress · N". This is the Zcash "visibly mark newly sent in a pending section" mandate. Rows here carry a live status chip + (where known) an ETA. They graduate down into the correct month section on completion.
- **Month sections.** One `section-header` per `Month YYYY` (current year may drop the year, e.g. "June"), each followed by its rows newest-first. Sections collapse-free (always expanded), continuous scroll.
- **Activity row (the core component).** Left→right:
  1. **Type icon** in a circular chip — distinct glyph per event type: Shield (shield-down / deposit), Private send (arrows / paper-plane), Unshield (shield-up / arrow-out), Bridge (bridge / link-across-chains). Color-class only carries semantic (incoming/outgoing/neutral) — designer picks hues.
  2. **Primary label** = human action: "Shielded", "Private send", "Unshielded", "Bridged in". Sub-line = a small **visibility glyph + word**: a closed-lock "Hidden" for private legs, an open-eye "Public" for public legs, or "Mixed" for shield/unshield (one leg each). For private send to a saved/known label show "Private send" only — never resolve a counterparty for a hidden row.
  3. **Right column** = amount with privacy-aware sign + asset code on the top line, status/time on the bottom line:
     - Shield: `+250.00 USDC` (credit into shielded balance).
     - Private send (outgoing): `−40.00 USDC`. Incoming private receive: `+40.00 USDC`.
     - Unshield: `−100.00 XLM`.
     - Bridge in: `+500.00 USDC` (neutral/public tint until shielded).
     - Bottom line: status chip when non-terminal (see states), else relative/clock time ("2:14 PM", "Jun 18").
- **Amount masking control (privacy affordance).** A small eye-toggle in the header (or inherited from home "hide balances") replaces all amounts with `••••` while preserving signs and asset codes. Shoulder-surfing defense; persists per session. Mark default = visible.
- **Row affordance.** Entire row is tappable → Screen 3 (detail). Hover/press state per design system.

### States

**Empty (no activity yet).** Centered illustration + heading **"No activity yet"** + body: *"Your shielded deposits, private sends, withdrawals, and bridge transfers will show up here."* Primary action button **"Add funds"** (→ shield / bridge area) and secondary text-link **"Receive privately"** (→ Receive). This is the teaching empty state, not Freighter's bare "No transactions to show" (borrow the branch, upgrade the copy).

**Empty — filtered to nothing.** When filters exclude everything: "No activity matches these filters." + **"Clear filters"** button. Distinct from the cold empty state (do not show "Add funds" here).

**Loading (first load).** Skeleton: 1 month-header bar + 4–6 shimmer rows (icon circle + two text bars + right-aligned amount bar). Reuse Freighter `components/Loading` scaffold, restyled to row skeletons. No spinner-only screen.

**Loading (incremental / pagination).** Bottom **"Load older activity"** footer button OR auto-load on scroll with a small inline spinner + "Loading older…". Never blank the already-rendered list.

**Refreshing (background sync).** Thin top progress line or a subtle "Syncing…" pill near the header while re-decrypting notes / polling chain; existing rows stay interactive. Mirrors Zashi's visible sync progress.

**Pending-proof (per-row, lives in the In-progress tray).** Status chip **"Proving…"** with an indeterminate micro-progress; sub-text "Generating zero-knowledge proof — keep this tab open." For unshield/private-send the chip may carry the proof validity hint ("valid ~3 min"). This is a FIRST-CLASS state, never a frozen spinner — the row is fully rendered with all known fields, only status is provisional.

**Attesting (bridge-specific, In-progress tray).** Chip **"Attesting…"** + sub-text "Waiting for Circle attestation (~minutes)." Two-chain mini-progress (ETH ✓ → Circle … → Stellar). Public-leg badge on the row.

**Awaiting-spendable (private receive / post-shield).** Chip **"Pending balance"** — funds arrived but not yet spendable (Railway "Pending → Spendable" model). Sub-text "Becoming spendable…". Graduates to a normal completed row + the home balance moves from Pending to Spendable.

**Submitting / broadcasting.** Chip "Submitting…" between proof-done and chain-confirmation. For unshield, an honest micro-note: "Submitted from <gG…public account> (visible)."

**Success / complete.** Terminal row: no chip, shows clock time. A brand-new just-completed row may get a one-time subtle highlight/flash on first render after it graduates from the tray (no persistent decoration — anti-slop).

**Failed.** Chip **"Failed"** (alert glyph, borrow Freighter `ActionIcon` "failed" → `Icon.AlertCircle`). Row stays in the list (Zcash: do not delete). Sub-text gives the cause class: "Proof failed", "Rejected on-chain", "Attestation failed", "Submission failed". Tapping → detail with retry where safe.

**Expired (proof/quote expiry).** Chip **"Expired"** for a proof whose validity window lapsed before submission, or a bridge quote that timed out. Zcash mandate: visibly mark expired, notify, do NOT delete. Detail offers **"Try again"** (regenerate proof) — never silently retry.

**Disabled / gated (mainnet privacy gate).** When network = mainnet and the privacy pool is gated, the In-progress tray and shielded rows still render historical data, but any inline retry/regenerate action that would create a NEW private tx is disabled with tooltip "Shielded transfers are testnet-only for now." Plain reads remain.

### Interactions
- Tap row → Screen 3 detail (slide-up modal on narrow widths, side-panel on wide — borrow Freighter `SlideupModal` usage).
- Tap **Filter** → Screen 2 sheet.
- Tap header eye-toggle → mask/unmask all amounts.
- Pull-to-refresh / refresh affordance → background sync.
- "Load older activity" → paginate.
- In-progress rows poll & live-update their chip in place (no list reflow jump — reserve the row height).
- Each `data-*`-marked surface is mocked: add a discreet **"Mock data"** chip in the header for the prototype.

### Mock data (fixtures)
- **In progress:** (a) Private send −40.00 USDC, "Proving… valid ~3 min"; (b) Bridge in +500.00 USDC, "Attesting… ETH ✓ · Circle …"; (c) Shield +250.00 USDC, "Pending balance".
- **June:** Shielded +250.00 USDC (Public source) · Private send −12.50 USDC (Hidden) · Unshielded −100.00 XLM (Mixed, submitted-from note) · Failed private send −5.00 USDC ("Proof failed").
- **May:** Bridged in +500.00 USDC (complete) → followed by Shielded +500.00 USDC (the one-tap shield of that bridge) — show them as two adjacent rows to teach the "bridge ≠ private, you shield after" two-step.
- **April:** Private receive +40.00 USDC (Hidden) · Unshielded −60.00 USDC (Expired proof, "Try again").

## Screen 2 — Filters sheet

**Purpose.** Narrow the list by event type, asset, status, and direction without leaving the privacy-honest framing. Keep it short — we have a fixed, small taxonomy (no arbitrary tokens, no multi-chain).

**Borrow from.** No first-class Freighter filter view; borrow the `SlideupModal` container + segmented/checkbox controls from the design system. Conceptually the "Hide dust" toggle (`settingsSelector.isHideDustEnabled`) already lives in Freighter history settings → we generalize it.

### Layout & components
- Slide-up sheet titled "Filter activity" with a close affordance and a **"Reset"** text action (top-right) that clears to defaults.
- **Type** — multi-select chips: Shield · Private send · Unshield · Bridge. Default = all on.
- **Asset** — segmented: All · XLM · USDC. (Fixed asset set — no search, no import.)
- **Status** — multi-select chips: In progress · Complete · Failed · Expired.
- **Visibility** — segmented teaching control: All · Hidden (private) · Public legs. This is product-specific and doubles as education.
- **Direction** — segmented: All · In · Out.
- **Hide dust** — toggle, "Hide transfers under a tiny amount" (carry Freighter's behavior).
- Sticky footer **"Show results (N)"** primary button with a live count; secondary "Cancel".

### States
- **Default:** all-inclusive; footer count = total.
- **Active filters:** applied filters reflected as a count badge on the Activity header funnel icon; "Reset" enabled.
- **Zero results preview:** footer shows "Show results (0)" disabled-style with helper "No activity matches — try widening." (prevents a dead-end apply).
- **Disabled:** on a brand-new wallet with empty activity, Filter button is disabled with tooltip "Nothing to filter yet."

### Interactions
- Selecting any control live-updates the footer count (computed over the mocked fixture).
- "Show results" applies + closes; chips persist for the session.
- Filter state is reflected in the funnel badge and is clearable from both here and the "Clear filters" empty state on Screen 1.

### Mock data
- Default count e.g. "Show results (12)". Selecting Type=Private send + Status=Failed → "(1)". Selecting Visibility=Public legs → bridge + shield-source + unshield-destination rows only.

## Screen 3 — Activity detail

**Purpose.** The single most important teaching surface in the cluster: show **the user's own decrypted truth** for the event AND an explicit, honest **disclosure block** ("what the world can see about this") with explorer links for public legs only. One layout, four type-variants.

**Borrow from.** `components/accountHistory/TransactionDetail/index.tsx` — title-row (`rowIcon` + title + dated subtitle "Jun 18 • 2:14 PM"), the credit/debit "Received/Sent" rows (`AssetDiff__row` with `ArrowCircleDown`/`ArrowCircleUp`), the **metadata block** (`Metadata` rows: Status with success/failed class, Fee, Memo via `TruncatedMemo` — "show the memo even if empty" per Zcash), and the bottom **"View on stellar.expert"** secondary button (`Icon.LinkExternal01`). We extend, not replace.

### Layout & components (top → bottom)
1. **Title row.** Type icon + title ("Private send", "Shielded deposit", "Unshield to Stellar", "Bridge from Ethereum") + dated subtitle (date • time) — Freighter title-row 1:1. A **visibility badge** sits beside the title: closed-lock "Hidden" / open-eye "Public" / "Mixed".
2. **Your view (decrypted) block.** Credit/debit rows (borrow `AssetDiff__row`): amount + asset, and a To/From line. CRITICAL per-type rules:
   - **Private send:** show your decrypted amount, asset, memo, and — only if the recipient is a contact you saved — a local label; otherwise "Recipient (private)". Never an on-chain identity.
   - **Shield:** show source = your **public Stellar account** (this leg IS public) → into shielded balance.
   - **Unshield:** show destination = the **public Stellar account** + amount; note source = "from your shielded balance (which deposit is hidden)".
   - **Bridge:** show Ethereum source (public) → Stellar mint (public), amount.
3. **Disclosure block — "What others can see".** The novel, mandatory panel. A two-column honest table:
   - Private send (Hidden): "Anyone on-chain can see: *that a shielded transaction occurred* and *a spent marker*. They cannot see: sender, recipient, amount, asset, or memo." No explorer link.
   - Shield (Mixed): "Public: your Stellar account + the deposited amount/asset. Hidden: nothing further — deposits are public by design." Explorer link to the **public deposit** only.
   - Unshield (Mixed): "Public: destination account, amount, asset, a spent marker, and the **submitting account** (no relayer yet — this links the withdrawal to that account). Hidden: which deposit it came from." Explorer links to destination tx + the submitting account; an honest warning line about the metadata leak.
   - Bridge (Public): "Public on both chains: Circle is the attester. This is an on-ramp, not a private transfer — shield it afterward to make it private." Explorer links to ETH burn + Stellar mint.
4. **Metadata block** (borrow Freighter `Metadata` rows): **Status** (Complete/Proving/Attesting/Pending balance/Failed/Expired with semantic class), **Network fee** (XLM, tiny/flat — no editor), **Memo** (private legs only; show even if empty per Zcash; clearly "encrypted, only you and the recipient can read this"), **Created** timestamp, and for in-progress: **Proof validity** / **ETA**.
5. **Footer actions (contextual).**
   - Public legs: **"View on stellar.expert"** per public link (multiple buttons for unshield/bridge). Disable + tooltip on custom/unsupported network (Freighter gates this with `isCustomNetwork`).
   - Hidden private send: NO explorer button; instead **"This transaction is private — there is nothing public to view."**
   - Failed/Expired: **"Try again"** (regenerate proof / re-quote) where safe.
   - Always: **"Generate disclosure proof"** → Screen 4 (the compliance hook, scoped to THIS one transaction — Penumbra transaction-perspective model, not a blanket view key).

### States
- **Complete (per type).** Full decrypted view + disclosure block + explorer links (public legs only).
- **Pending-proof.** Title-row renders, "Your view" shows intended amounts greyed as "Pending", Status="Proving…", footer shows live progress + "keep tab open"; explorer buttons absent until a public leg exists. No frozen spinner.
- **Attesting (bridge).** Two-chain stepper (ETH burn ✓ → Circle attesting… → Stellar mint pending), ETA, public-leg links appear as each leg lands.
- **Awaiting-spendable.** Status="Pending balance", helper "These funds aren't spendable yet."
- **Failed.** Status="Failed" (failed class), reason ("Proof failed" / "Rejected on-chain" / "Attestation failed"), "Try again" enabled where safe; disclosure block reflects that nothing public happened (for a pre-submission proof failure) or that a public attempt is visible (post-submission).
- **Expired.** Status="Expired", "Try again" (regenerate). Honest note that the prior proof/quote is no longer valid.
- **Disabled.** On mainnet-gated, "Try again"/"Generate disclosure proof" disabled with reason tooltip; read-only detail still shown.
- **Loading detail.** If opened before metadata hydrates (deep link), skeleton the blocks; never block the whole modal on the explorer-link fetch.
- **Copy success.** Copying an address/tx-hash → inline "Copied!" (borrow Freighter CopyText pattern).

### Interactions
- Copy any public address / tx hash (inline "Copied!").
- Open explorer (new tab) — public legs only; honest label that this reveals you're looking at a public tx.
- "Generate disclosure proof" → Screen 4.
- "Try again" → routes back into the relevant Shield/Send/Unshield/Bridge flow pre-filled.
- Close → back to list (modal dismiss).

### Mock data
- Private send detail: −12.50 USDC, memo "rent split" (encrypted), Recipient (private), Status Complete, Fee 0.00001 XLM, "Nothing public to view".
- Unshield detail: −100.00 XLM → `GDEST…X4QK`, submitting account `GSUB…7AOO` (leak warning), spent marker, two explorer links.
- Bridge detail: +500.00 USDC, ETH burn `0xabc…`, Circle attestation id, Stellar mint `GMINT…`, two-chain stepper complete.

## Screen 4 — Per-transaction disclosure proof (compliance hook)

**Purpose.** Let the user *selectively* prove ONE transaction to a third party (auditor, accountant, exchange) without handing over a wallet-wide viewing key. Penumbra transaction-perspective model (refs: "Compliance / selective disclosure (BETTER model)"). Lives off the detail screen; the wallet-wide view-key export lives in Settings (out of this cluster) — link to it but warn it's broader.

**Borrow from.** No direct Freighter reference (novel). Borrow the share/copy + blunt-warning patterns from Railway view-only-key copy ("cannot be revoked", "do not paste into email") and the export/confirmation scaffolding generally.

### Layout & components
- Sheet titled "Generate disclosure proof" with the target tx summarized at top (icon, type, amount, date — read-only).
- Explainer: *"This proves the details of this single transaction to whoever you share it with. It does not reveal any of your other activity."* Contrast callout: *"This is safer than your wallet viewing key, which would reveal all past and future activity."* with a link "About viewing keys" (→ Settings).
- **What will be revealed** checklist (per type): amount, asset, memo, counterparty (if you choose to include), timestamp — each a toggle so the user controls granularity. Default = minimum needed.
- Output options: **Copy disclosure proof**, **Download (.json/.txt)**, **Show QR** (for in-person). Blunt warning line: "Anyone with this proof can verify this transaction's details. It cannot be un-shared."
- Mocked-integration banner: **"Disclosure proof generation is mocked in this prototype."**

### States
- **Default / configuring:** toggles set, primary "Generate" enabled.
- **Generating (pending-proof):** "Generating proof…" honest progress (this too is a client-side ZK op) — keep-tab-open note; not a frozen spinner.
- **Ready / success:** proof shown (truncated, copyable) + Copy/Download/QR + the irrevocability warning; "Copied!" on copy.
- **Error:** "Couldn't generate proof" + Retry.
- **Disabled (mainnet-gated):** primary disabled with reason; explainer still readable.

### Interactions
- Toggle reveal granularity → preview of the "What will be revealed" summary updates live.
- Generate → Ready; Copy/Download/QR.
- Link to Settings view-key export (with the "broader, cannot be revoked" warning).

### Mock data
- Target: Private send −12.50 USDC, Jun 18. Toggles: amount ✓, asset ✓, memo ✓, counterparty ✗, timestamp ✓. Output: a mocked base64-ish proof string + a QR. Warning surfaced.

## Cross-cluster rules (apply to every screen above)
- **Honest framing everywhere.** Never "fully private"/"anonymous". Private send = "hidden from others"; shield/unshield/bridge legs explicitly labeled "public". The unshield submitting-account metadata leak is always disclosed (no relayer today).
- **Pending is first-class, never a frozen spinner.** Proving / attesting / pending-balance / submitting / awaiting-spendable each have named chips, ETAs where known, and "keep tab open" guidance. Rows render fully with provisional status.
- **Privacy glyph discipline.** Closed-lock = Hidden, open-eye = Public, "Mixed" = both legs. Consistent across list rows, detail badge, and filter "Visibility" control.
- **No explorer link for hidden legs.** Public legs only; each explorer action is honestly labeled as viewing public data.
- **Don't delete failed/expired rows.** Mark and keep (Zcash mandate); offer "Try again", never silent retry.
- **Network-scoped + gated.** Activity reads the active network; mainnet privacy gate disables NEW-tx actions (retry/disclosure) with a reason, read stays.
- **Everything mocked.** Header "Mock data" chip on Screen 1; "mocked in this prototype" banner on Screen 4; bridge marked "to-be-validated on testnet".
- **No DO-NOT-COPY creep.** No swaps row, no fiat buy/sell, no NFT history, no dApp-connect log, no custom-gas, no arbitrary token rows. Assets are XLM + USDC only; "trustline" never appears.
