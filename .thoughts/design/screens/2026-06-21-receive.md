# Receive — screen cluster spec

> Product: **ZK Fighter** (brand locked). A privacy-by-default, self-custody zero-knowledge wallet for shielded payments on Stellar. Browser web app first.
>
> Cluster owner: Receive. Sibling clusters (Home, Send, Shield/Unshield, Add-funds/Bridge, Settings, Compliance) referenced where they hand off — not specified here.
>
> Primary reference: `reference/freighter/extension/src/popup/views/ViewPublicKey/index.tsx` (maps ~1:1 to the public-address half). Privacy layering invented from spec + Zashi/Railgun framing. QR specifics from SEP-0007 + qrcode.com capacity docs. See `.thoughts/design/2026-06-21-design-references.md`.
>
> **Mock scope:** every surface here renders from MOCKED wallet data + a MOCKED "register / publish keys" directory call. Mark each mocked surface inline with a small `MOCK` chip in the prototype.

## 0. Cluster mental model (the thing this cluster must teach)

The single hardest idea in the whole product lives on this screen: **one wallet, two addresses, two very different privacy meanings.** Receive is where the user first confronts it, so every screen here is a teaching surface, not just a copy-button.

- **Private address** (the headline). A single bundled string = note key (BN254) + encryption key (X25519), Railgun-0zk style, prefixed (e.g. `zkf1…`). Payments to it land in the **shielded** pool. This is what you hand to a person who is paying you and using this app.
- **Public Stellar address** (the plumbing). A normal `G…` account. Used only for deposits, on-ramp/bridge landings, and as the fee/relay account. Anything that touches it is **public** on Stellar.
- The two are **the same wallet, same seed** — reassure relentlessly. The "wrong address" mix-up is a documented footgun; the design's job is to make the private one obviously primary and the public one obviously "funding only," while never letting the user think they are two separate wallets or two separate balances.

Honest-framing rules enforced across the cluster:
- Never say "anonymous" or "fully private." Say **"shielded"** / **"private payments."**
- The public address screen must say, in plain words, that **deposits to it are public** and only become private once shielded.
- The private address itself is safe to share (it's a receive key, not a spend key) — say so, because users will be scared to paste a long crypto string publicly.

## Screen inventory

1. **R1 — Receive (Private tab, default)** — the primary receive surface.
2. **R2 — Receive (Public / Deposit tab)** — the Stellar `G…` half for deposits & on-ramp.
3. **R3 — QR fullscreen / "Show to scan"** — enlarged scannable code (shared by both tabs).
4. **R4 — Register in directory (publish keys) sheet** — optional discoverability opt-in + confirm + manage.
5. **R5 — Request a specific amount (private payment request)** — optional pre-filled amount/asset/note → shareable link/QR.
6. **R6 — Address-revealed / first-time teaching interstitial** — one-time "same wallet, two addresses" explainer gating first Receive open.

## R1 — Receive · Private tab (DEFAULT)

### Purpose
Give the user the string + QR to **get paid privately**, framed so they trust pasting it publicly, understand it routes to their shielded balance, and don't confuse it with the public deposit address.

### Layout & components (top → bottom)
1. **App header** — back chevron (returns to Home), centered title `Receive`. Right slot: kebab/overflow → `Request an amount` (opens R5), `Register so people can find you` (opens R4), `Show public deposit address` (jumps to R2). *(Reuse the `View.AppHeader` pattern from ViewPublicKey; replace the inline account-rename with a static title since rename lives in Accounts here.)*
2. **Segmented tab control** — two segments, full width: **`Private`** (active) · **`Public deposit`**. A small lock/shield glyph on Private, a bank/deposit glyph on Public. Persistent helper line directly under the segments: *"Same wallet — two addresses for two jobs."* (This is the anti-confusion anchor; it never disappears.)
3. **Privacy badge row** — a compact, non-decorative status pill: `Shielded · payments here stay private`. This is the semantic counterweight to R2's public badge; the two badges must read as opposites at a glance.
4. **QR card** — the bundled private string encoded as a QR.
   - Rendered **physically larger** than a normal address QR (private string is long → high QR version), high-contrast, full quiet-zone, **EC level M (auto-drop to L if it overflows a comfortable size)**, **never a logo overlay** (logo forces EC H and over-densifies a long payload).
   - Do **NOT** wrap in a SEP-0007 `web+stellar:pay` URI — that scheme is only for `G…` accounts. The QR payload is the raw bundled private-address string (optionally our own `zkf://receive?addr=…` deep-link wrapper for app-to-app scan; decide at build, but it is NOT SEP-0007).
   - Tap/click QR → opens **R3** fullscreen.
   - *(Borrow `QRCodeSVG` usage + `.qr-code` container from ViewPublicKey; override size + EC level.)*
5. **Address display** — label `Your private address`, then the string shown **middle-truncated with head+tail preserved** for a visual checksum (e.g. `zkf1qg…7f4a`). A subtle "checksum" affordance: show the last 4 chars in a slightly emphasized weight so the user can verbally confirm "ends in 7f4a." Full string is in the copy payload, never forced on screen as a wall of characters.
6. **Primary action — `Copy private address`** — full-width primary button. Copy is the PRIMARY action (manual typing of this string is infeasible). On success → `Copied!` confirmation (button label swap + checkmark, ~1.5s), then reverts. *(Borrow `CopyText` + `doneLabel="Copied!"` exactly from ViewPublicKey.)*
7. **Secondary actions row** — `Show to scan` (→ R3) · `Request an amount` (→ R5). Tertiary/rounded styling.
8. **Reassurance / teaching block** (collapsible, expanded by default on first 3 views, then collapsed with a "Why is this safe to share?" toggle):
   - *"This is a receive-only address. Sharing it lets people pay you — it can't be used to spend your funds or drain your wallet."*
   - *"Payments to this address are shielded: the amount and your balance stay private."*
   - *"It's the same wallet as your public Stellar address — money you receive here shows up in one balance."*
9. **Footer link** — `Learn how private payments work` (opens the privacy-model explainer; same content as R6). No explorer link on this tab (a private address has nothing to show on a public explorer — and saying so is itself a teaching moment: a one-line note *"Private payments don't appear on the public explorer."*).

### States
- **Default / loaded** — as above. Mock string present, QR rendered, copy enabled.
- **Loading** — while wallet/keys hydrate: QR card shows a skeleton block (not a frozen spinner), address line shows a shimmer, copy button disabled. Mirror Freighter's `Loading` component placement. *(Keys are local/deterministic so this is fast, but still design it.)*
- **Empty / not-yet-generated** — should essentially never happen (the private address is deterministic from seed and exists the moment onboarding completes). If keys are somehow unavailable: friendly state *"Your private address will appear once your wallet finishes setting up."* + `Retry`. NOT an error red state.
- **Error** — key derivation/read failed: `Notification variant=error` — *"Couldn't load your private address."* + `Try again`. Copy/QR hidden while errored. *(Borrow the error `Notification` block from ViewPublicKey lines 75–86.)*
- **Disabled (network-gated)** — Receiving a private payment requires the privacy pool to exist on the active network. On **mainnet** (privacy pool not deployed today), show the address but add an inline caution: *"Private receive is testnet-only right now. On mainnet, share your public deposit address instead."* with a one-tap `Switch to Testnet` and a deep-link to R2. Copy stays enabled (the address is still valid for when the pool ships) but the "stays private" badge is muted with the caveat. NEVER silently imply privacy that isn't available.
- **Success (copied)** — transient `Copied!` on the button + a brief toast *"Private address copied"*. The reassurance block surfaces *"Paste it anywhere — it's safe to share."* once, the first time the user copies.
- **Pending-proof** — N/A for displaying a receive address (no proof to receive). BUT: if the user arrives here right after a Shield/Send that is still proving, show a small dismissible banner at top: *"A shielded transfer is still finishing (~Xs). New incoming payments are unaffected."* linking to Activity — so the proving state from elsewhere is acknowledged, not hidden.
- **Registered vs not-registered** — if the user has published keys (via R4), show a tiny inline chip near the address: `Listed in directory` with a `Manage` link (→ R4 manage mode). If not registered, the kebab/secondary still offers `Register so people can find you`.

### Interactions
- Copy → clipboard write → `Copied!` swap + toast. On clipboard-API failure (insecure context / permission), fall back to selecting the full string in a revealed read-only field with *"Copy failed — select and copy manually."*
- Tap QR → R3.
- Tab switch to `Public deposit` → R2 (animated slide, both tabs retained — borrow Freighter Send's in-place persisted-step slider feel).
- Toggle reassurance block; toggle `Show full address` (reveals the untruncated string in a selectable, wrapped monospace field for power users who want to eyeball every char).

### Mock data
- Private address: `zkf1qg9x7m2k4p…d3v8n6r0a1c5e9b2…7f4a` (full ~120–180 char mock; truncated display `zkf1qg9x7m…7f4a`).
- Network: `Testnet` (badge). Directory status: `not registered` by default; a second mock profile flips to `Listed`.

### Borrow from
- `reference/freighter/extension/src/popup/views/ViewPublicKey/index.tsx` — QRCodeSVG block, address-copy-label + truncated address + `CopyText doneLabel="Copied!"`, header layout, error `Notification`, footer button slot.
- `reference/freighter/extension/src/popup/components/CopyValue/index.tsx` — inline copy-icon + value affordance for the "show full address" field.
- Framing/microcopy: Zashi shielded-address reassurance + Railgun shield framing (`.thoughts/design/2026-06-21-design-references.md` privacy table rows "Receive — private + public" and "Shield").

## R2 — Receive · Public deposit tab

### Purpose
Give the `G…` Stellar address + a SEP-0007 deposit QR for **funding** (exchange withdrawals, on-ramp, bridge landings, a payer who can only send to a normal Stellar address). Make unmistakably clear that **anything landing here is public** until shielded.

### Layout & components
1. **App header** — same as R1.
2. **Segmented control** — `Private` · **`Public deposit`** (active). Same persistent helper line *"Same wallet — two addresses for two jobs."*
3. **Public badge row** — pill: `Public · visible on Stellar`. Visually the inverse of R1's shield pill (this contrast is the whole lesson).
4. **Honest-framing callout** (always visible, not collapsible) — *"Deposits to this address are public. To make them private, shield them — we'll prompt you with one tap when funds arrive."* Includes a `What is shielding?` link.
5. **QR card** — encodes a **SEP-0007 `web+stellar:pay` URI**, not the bare `G…` key:
   - `destination` = public `G…`; `asset_code`/`asset_issuer` available for XLM and USDC (default XLM, asset switch in R5-style amount flow); `amount` omitted on the plain receive QR; **`network_passphrase` set to the active network** so a Testnet QR can't be paid on Mainnet.
   - Standard size/EC (payload is short) — EC level M, optional small brand mark acceptable here (short payload tolerates EC H if a logo is ever wanted, but keep parity with R1: no logo by default).
   - Tap → R3 fullscreen (R3 shows whichever tab's payload is active).
6. **Address display** — label `Your public Stellar address`, middle-truncated `G…` keeping head+tail (e.g. `GА7K…7AOO`) for checksum confirmation, matching Freighter exactly.
7. **Primary action — `Copy Stellar address`** → `Copied!`.
8. **Asset/trustline status (silent-trustline surface)** — a quiet status line, NOT the word "trustline":
   - If USDC is ready to receive: `✓ Ready to receive XLM and USDC`.
   - If USDC isn't set up yet: `Preparing to receive USDC…` with an auto-handled background setup (we silently establish the user's own USDC trustline at first-receive). If it costs reserve, surface only the plain-language cost: *"Receiving USDC reserves 0.5 XLM in your account (one-time)."* — never "trustline," never a contract id. `MOCK` chip.
   - Error variant: *"USDC receiving isn't ready yet."* + `Retry` (mock).
9. **Unfunded-account empty state** (Stellar-specific, critical) — a brand-new `G…` is inert ("does not exist") until funded with ≥1 XLM. If unfunded:
   - Friendly empty block, **not an error**: *"This address isn't active yet. Send it at least 1 XLM to start using it."* + `Learn why` (min-balance footnote: base reserve 0.5 XLM, 1 XLM locked on a bare account).
   - **Testnet only:** one-click `Fund with Friendbot` (`MOCK` → simulates a funded balance). On mainnet, no Friendbot — show the "send 1 XLM" guidance only.
10. **Footer** — `View on stellar.expert` button (active-network explorer link) — borrowed verbatim from ViewPublicKey footer (lines 200–218). This link is meaningful here (public address) and pointedly absent on R1.

### States
- **Default / funded** — address + SEP-0007 QR + ready-to-receive status + explorer link.
- **Loading** — skeleton QR + shimmer address, copy + explorer disabled.
- **Empty / unfunded** — the unfunded empty state (#9). Explorer link still works (shows "account not found" gracefully via stellar.expert) but we pre-empt with our own friendly copy.
- **Error** — account data fetch failed → error `Notification` (verbatim pattern from ViewPublicKey 75–86): *"Failed to fetch your account data."*
- **Disabled** — none for copy; if network indicator is mid-switch, dim the QR briefly with *"Updating for {network}…"* (because the SEP-0007 `network_passphrase` and USDC issuer change with the toggle).
- **Success (copied)** — `Copied!` + toast *"Stellar address copied"*.
- **Pending-proof** — N/A (public address). If the user just shielded a deposit and proving is underway, show the same dismissible *"A shielded transfer is finishing…"* banner as R1.
- **USDC-setup pending** — the `Preparing to receive USDC…` inline state (auto-resolves; mock 2s).

### Interactions
- Copy `G…` → `Copied!`.
- `Fund with Friendbot` (testnet) → mock funds → transitions empty→funded with a success toast.
- `Switch to {network}` if arrived from R1's mainnet-gated state.
- Tap QR → R3.
- `What is shielding?` / `Learn why` → explainer (R6 content or min-balance doc).
- Tab switch → R1.

### Mock data
- Public address: `GA7K3JQ2M9X4P6R8S1T0U2V4W6X8Y0Z2A4B6C8D0E2F4G6H8I0J7AOO` → truncated `GA7K…7AOO`.
- SEP-0007 QR payload (mock): `web+stellar:pay?destination=GA7K…7AOO&network_passphrase=Test%20SDF%20Network%20%3B%20September%202015`.
- Balance: funded variant `12.5 XLM, 0 USDC`; unfunded variant `0 / account not found`.

### Borrow from
- `ViewPublicKey/index.tsx` — entire QR + truncated-address + Copy + explorer footer scaffold (this tab is the closest 1:1 to Freighter; the privacy callouts are the only additions).
- Unfunded empty-state framing + Friendbot — `.thoughts/design/2026-06-21-design-references.md` row "Shield/deposit — unfunded empty state."

## R3 — QR fullscreen ("Show to scan")

### Purpose
Maximize scannability for in-person payment — enlarge the active tab's QR so a phone camera reads a long private string reliably, while keeping the user oriented on WHICH address they're showing.

### Layout & components
1. **Dim/scrim overlay** over the wallet, centered modal/sheet.
2. **Context header** — `Showing your PRIVATE address` or `Showing your PUBLIC deposit address`, with the matching shield/public badge so the user (and the scanner) can't mistake which one is on screen. This label is load-bearing: showing the wrong QR is the footgun.
3. **Large QR** — maximum comfortable size, full quiet zone, brightness-boost hint (the prototype can include a "screen brightened for scanning" note; real impl raises screen brightness). EC level M (private) — drop to L only if it overflows; never a logo on the private QR.
4. **Truncated address line** beneath, head+tail, for verbal/visual confirmation.
5. **Actions** — `Copy` (same payload), `Close`. If on Private, a `Switch to public` toggle and vice-versa, so a payer who "can only send to a G… address" is one tap away.

### States
- **Default** — enlarged QR for active tab.
- **Loading** — brief skeleton (instant in practice).
- **Error** — if QR can't encode (payload too long even at EC L — should not happen for our string sizes, but guard it): *"Can't display a scannable code — copy the address instead."* + `Copy`.
- **Disabled (mainnet private)** — if private receive is network-gated, R3 for the private tab shows the QR with a caption *"Testnet only — see public deposit on mainnet."*
- **Success** — copy toast.
- **Pending-proof / empty** — N/A.

### Interactions
- Tap scrim or `Close` → back to R1/R2. `Switch to public/private` → swaps payload + relabels. `Copy`.

### Mock data — reuses R1/R2 payloads.

### Borrow from
- No direct Freighter fullscreen-QR ref (Freighter shows QR inline only) — extend ViewPublicKey's `QRCodeSVG` into a modal. Sizing/EC guidance from `2026-06-21-design-references.md` QR section + qrcode.com capacity tradeoff.

## R4 — Register in directory (publish keys) sheet

### Purpose
OPTIONAL discoverability: let a user publish their private-address keys to a lookup directory so others can find/pay them by a handle, instead of pasting a long string. This is a privacy *trade* — publishing links a human-readable handle to receive keys — so it must be explicit opt-in with an honest cost statement, and reversible.

### Layout & components
1. **Sheet header** — `Get found by a handle` + subtitle *"Optional. Let people pay you without copying your long address."*
2. **What gets published (honest disclosure block)** — plain bullets:
   - *"Your private RECEIVE address (note + encryption keys) is published. People can pay you privately."*
   - *"Your chosen handle is publicly linked to that address — so this address is no longer pseudonymous."*
   - *"It does NOT publish your balance, your transactions, or any spend ability."*
   - *"You can unpublish anytime, but anyone who already saved your handle keeps the address."*
3. **Handle input** — `@handle` field with live availability check (`MOCK`): debounce → `Checking…` → `Available ✓` / `Taken ✗` / `Invalid (use a–z, 0–9, _)`.
4. **Preview** — `people will pay: @yourhandle → (resolves to your private address)`.
5. **Primary CTA** — `Publish my address`. Secondary `Not now`.
6. **Manage mode** (when already registered) — shows current `@handle`, `Copy share link` (`zkf.app/@handle`, mock), `Unpublish` (destructive, confirm dialog: *"People can no longer find you by @handle. Already-saved links keep working until they refresh."*).

### States
- **Default (not registered)** — disclosure + empty handle field; Publish disabled until a valid+available handle.
- **Loading** — availability `Checking…`; on submit, button → `Publishing…` (spinner), inputs locked.
- **Empty** — handle blank → CTA disabled with helper *"Pick a handle to continue."*
- **Error** — availability check failed → *"Couldn't check that handle. Try again."*; publish failed → *"Publishing didn't go through."* + `Retry`. (`MOCK` directory.)
- **Disabled** — `Publish` disabled while handle invalid/taken/empty; whole sheet disabled-with-reason on mainnet if directory is testnet-only (*"Directory is testnet-only for now."*).
- **Success** — confirmation: *"You're listed as @yourhandle"* + `Copy share link` + `Done`. R1 now shows the `Listed in directory` chip.
- **Pending-proof** — N/A (directory write is a normal signed Stellar/off-chain op, not a ZK proof). If it does require an on-chain publish tx, show normal `Submitting…`, not a proof step.

### Interactions
- Type handle → debounced availability. Publish → success → R1 chip updates. Unpublish → confirm → chip removed.

### Mock data — handle `@nova`; `@nova` taken, `@nova_x` available; share link `zkf.app/@nova_x`.

### Borrow from
- No Freighter ref (Freighter has no directory). Borrow sheet/CTA + destructive-confirm patterns generically; honest-disclosure tone from the view-key warning microcopy (Railgun "cannot be revoked once shared" cadence, adapted to "already-saved links keep working").

## R5 — Request a specific amount (private payment request)

### Purpose
Let the user generate a **pre-filled private payment request** — address + amount + asset (+ optional short note) → a shareable link and a QR — so a payer scans and the amount is filled in. Keeps memo minor (not a headline feature).

### Layout & components
1. **Header** — `Request a payment`.
2. **Asset toggle** — `XLM` / `USDC` segmented.
3. **Amount input** — numeric, big, with asset suffix; optional (a request with no amount = "pay me any amount").
4. **Note field (minor)** — short optional text, clearly marked *"Only you and the payer see this; it's part of the encrypted payment."* De-emphasized.
5. **Destination summary** — read-only: routes to `your private address (zkf1qg…7f4a)` with the shield badge.
6. **Generated request** — QR + `Copy request link` (`zkf.app/pay?to=…&amt=…&asset=…`, mock) + `Show to scan` (R3 variant showing this enriched payload).
7. **Primary** `Create request`; the QR/link appear after creation.

### States
- **Default** — empty amount, XLM selected, note empty, Create enabled (amount optional).
- **Loading** — `Creating…` while building the link/QR (instant; still designed).
- **Empty** — no amount → request is "any amount"; show helper *"No amount set — payer chooses."*
- **Error** — invalid amount (≤0, too many decimals — XLM 7dp, USDC standard) → inline field error; link-gen failure → retry.
- **Disabled** — `Create` disabled on invalid amount; whole flow disabled-with-reason on mainnet private-gating (reuse R1's network caveat).
- **Success** — QR + link shown + `Copied!` on link copy + toast.
- **Pending-proof** — N/A (creating a request is local; the PROOF happens on the payer's Send, not here).

### Interactions
- Switch asset → suffix + decimal rules update. Type amount → validate. Create → reveal QR+link. Copy link → `Copied!`. Show to scan → R3.

### Mock data — `25.00 USDC`, note `"Coffee"`, link `zkf.app/pay?to=zkf1qg…7f4a&amt=25&asset=USDC`.

### Borrow from
- Amount + asset-picker pattern from `reference/freighter/extension/src/popup/views/Send/index.tsx` (AMOUNT step + asset picker) — re-skinned as a request rather than a send. QR/copy from ViewPublicKey.

## R6 — First-time "same wallet, two addresses" interstitial

### Purpose
A one-time teaching gate the very first time the user opens Receive, so the private/public distinction is learned before they can copy the wrong one. After dismissal it never auto-shows again (re-openable via `Learn how private payments work`).

### Layout & components
1. **Two-card diagram** side by side: a `Private` card (shield) → "people pay you, stays private" and a `Public deposit` card (bank) → "fund from exchanges/on-ramp, public" — joined by a brace labeled **`one wallet · one balance · one seed`**.
2. **Three honest lines**:
   - *"Share your private address to get paid privately."*
   - *"Use your public Stellar address to deposit or on-ramp — those are public until you shield them."*
   - *"Both are the same wallet. Money arrives in one balance."*
3. **CTA** — `Got it, show my addresses` → R1. Secondary `Don't show this again` (checked by default after first dismissal).

### States
- **Default** — shown once. **Dismissed** — skip straight to R1 thereafter. No loading/error (static content). No empty/pending states.

### Interactions
- `Got it` → R1. The footer `Learn how private payments work` on R1/R2 reopens this content as a passive explainer (no gating).

### Mock data — none (static educational content).

### Borrow from
- No Freighter ref (Freighter has no privacy concept). Diagram framing from Zashi "shielded vs transparent, same wallet" reassurance (`2026-06-21-design-references.md` privacy table).

## Cross-cluster consistency notes (for the designer)

- **Badge opposition:** the `Shielded · private` pill (R1/R3) and `Public · visible on Stellar` pill (R2/R3) must be a deliberate visual pair — same shape/size, clearly contrasting semantics. This pairing is reused on Home (shielded vs spendable) and Activity (private vs public rows). Keep it consistent across clusters.
- **Truncation rule (global):** middle-truncate ALL addresses keeping head+tail for checksum; `zkf1…` and `G…`/`C…` truncate identically. Provide a `Show full address` reveal everywhere a truncated address appears.
- **Copy is primary on Receive** (manual entry infeasible for the private string); `Copied!` label-swap + toast is the standard success affordance (lifted from Freighter `CopyText doneLabel`).
- **Network indicator** is global (persistent TESTNET pill in the app header); Receive screens additionally caveat private-receive gating on mainnet rather than relying on the global pill alone.
- **No SEP-0007 on the private QR; SEP-0007 `pay` only on the public QR** — and only `pay`, never `tx` (that's signing, out of scope).
- **Mark every mocked surface** (`MOCK` chip): directory register/availability/share-link (R4), request-link generation (R5), Friendbot funding + USDC-ready status (R2), and any balance/funded-state toggles.
- **Anti-slop:** R1 and R2 must NOT look like two identical cards with different text — the privacy meaning has to be felt (badge, callout placement, explorer-link present/absent, QR size difference). Every Receive screen teaches the model; none is decoration-only.
