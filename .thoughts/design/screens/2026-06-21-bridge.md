# Screen cluster: Bridge / Add funds from another chain

> Product: **ZK Fighter** (brand locked). Privacy-by-default, self-custody ZK wallet for shielded payments on **Stellar**. Browser web app. Assets: **XLM + USDC**. Network: **testnet/mainnet config toggle** with a visible indicator; mainnet privacy may be gated.
>
> This cluster owns the **on-ramp**, not a transfer. It brings outside USDC onto Stellar so the user can then shield it. It is **two separate, non-atomic steps**:
> 1. **Bridge leg (PUBLIC, Circle CCTP):** USDC is **burned on Ethereum** → **Circle attests** (~minutes) → USDC **mints as PUBLIC USDC on Stellar**. The whole bridge leg is public; Circle is the attester. The funds arrive as ordinary, visible Stellar USDC.
> 2. **Shield leg (the privacy step):** the user taps **"Shield this"** to move the freshly-arrived public USDC into the shielded pool. This hands off to the **Shield flow** (`2026-06-21-shield-unshield.md`, PART A).
>
> Honesty rules that bind every screen here:
> - **The bridge is an on-ramp, never a private transfer.** Never imply the bridge hides anything. Both the Ethereum burn and the Stellar mint are public on-chain events; Circle sees the route.
> - **Two steps, not one.** Arrival ≠ shielded. The UI must always distinguish "arrived as public USDC" from "shielded". Privacy begins only after the shield leg.
> - **Cross-chain pending is multi-minute** (burn confirmations + attestation wait + Stellar mint). Design an honest, resumable, multi-step **timeline** — never a frozen spinner. The user can close the wallet and come back.
> - The word **"trustline" never appears.** USDC must be receivable on the Stellar side before the mint can land; set up / pre-check the user's USDC trustline invisibly (surface only the one-time +0.5 XLM reserve cost if actually charged).
> - **Mock everything.** Bridge integration is **"to-be-validated on testnet"** (a feasibility spike is still pending). Mark the whole cluster with a persistent **"Bridge — to be validated on testnet"** ribbon, and tag every simulated surface (attestation timer, explorer links, balances, Ethereum connection) with a small **"Mocked"** chip.
> - **No DEX / fiat / buy.** This replaces Freighter's "Buy with Coinbase" on-ramp; we keep only the cross-chain transfer mechanism, re-skinned as a USDC bridge-in.

## Cluster IA & shared scaffolding

**Entry.** Home → **"Add funds"** opens the Add-funds chooser (Freighter `views/AddFunds/index.tsx` — same two-button layout, re-skinned). The "Buy with Coinbase" button is **removed** (DO-NOT-COPY: fiat on-ramp). The two methods become:
- **"From another chain"** (this cluster) — bridge USDC from Ethereum.
- **"Transfer from another account"** (→ Receive, public Stellar address tab — `views/ViewPublicKey`).
Plus a tertiary line: **"Already have public funds? Shield them"** (→ Shield, `SOURCE` step).

**Flow shape.** Bridge borrows the **Send multi-step in-place slider** (Freighter `views/Send/index.tsx`): one route, N panels stacked, only the active panel visible, visited panels persist, directional slide (`from-right` forward, `dismiss`/`from-bottom` back), per-step back/close. Reuse the `STEPS` enum + `goToStep(next, anim)` + `visitedSteps` map structure verbatim.

Steps: **`SOURCE` → `AMOUNT` → `CONNECT` → `REVIEW` → `BURN` → `TRACKING` → `ARRIVED` → (hand off to Shield)**.
`TRACKING` is the long-lived one — it is **resumable** (re-entered from Home/Activity, not just inline) and persists across wallet close.

**Shared chrome borrowed from Freighter:**
- **`SubviewHeader`** (title + back chevron + close X) atop each step — `views/AddFunds/index.tsx`.
- **`View.Content`** scroll body + sticky footer button zone — `views/Send/index.tsx`.
- **`Notification`** (`@stellar/design-system`; `primary`/`warning`/`error`) for honest disclosures + error banners — used in `views/Send/index.tsx` for the simulate-fail case.
- **`Loading`** spinner for transient fetches — `views/AddFunds/index.tsx`.
- **`FullscreenSuccessMessage`** scaffold for the terminal success after shielding — `views/FullscreenSuccessMessage`.
- **`AccountHistory` row + month-section** structure for the resumable in-progress entry — `views/AccountHistory/index.tsx`.

**Cluster-specific shared primitives (invent, reuse across steps):**
- **`BridgeTimeline`** — the vertical 3-stage progress rail: **Burning on Ethereum → Circle attesting → Minting on Stellar**, each node with a state (pending / active / complete / failed) and a per-leg explorer link. The spine of `TRACKING`. Every node names the chain it lives on so the user always knows *where* their money is.
- **`ChainBadge`** — a small "Ethereum" / "Stellar" / "Circle" label with chain glyph, used on timeline nodes and amount rows so the cross-chain nature is never ambiguous.
- **`PublicLegBanner`** — recurring honest callout: "This bridge is **public**. The Ethereum burn, the Circle attestation, and the Stellar arrival are all visible on-chain. Privacy starts only when you **shield** after it arrives." (This is the cluster's teaching device; it is **not** the Shield/Unshield `PrivacyLedger` — different content, do not duplicate the same card.)
- **`MockedRibbon`** — persistent top ribbon "Bridge — to be validated on testnet · simulated", present on every step.
- **`AttestationClock`** — an honest elapsed/estimated timer for the attestation wait ("~13 min typical · 4:20 elapsed"), never a bare spinner.

**Cross-cluster handoffs:**
- On arrival, **"Shield this now"** routes into Shield `SOURCE` with USDC + amount pre-filled (`2026-06-21-shield-unshield.md` A1).
- Network toggle: the bridge mints to whatever network is active. On **testnet**, source chain is **Ethereum Sepolia**; on **mainnet**, gate per spec (bridge to-be-validated → show "Bridge runs on testnet only for now" disabled state on mainnet).

## B5. Burn (submitting on Ethereum) (`BURN`)

**Purpose:** The brief window where the Ethereum burn tx is being signed/submitted/confirmed. This is the first node of the timeline going live.

**Layout & components**
- `SubviewHeader` title "Starting bridge…", **no back** (irreversible once signed); a Close X that warns and routes to `TRACKING` rather than cancelling.
- **`BridgeTimeline`** appears here for the first time, node 1 (**Burning on Ethereum**) = active, nodes 2–3 pending.
- Status text under node 1: "Confirm the burn in your Ethereum wallet" → then "Waiting for Ethereum confirmations (1/3)…" with a live counter (mocked).
- Ethereum tx explorer link (mocked) once a hash exists.
- Reassurance: "You can leave this screen — it'll keep going."

**States**
- **Awaiting signature:** "Confirm in your Ethereum wallet" + `Loading`. Cancel possible **only before signature** (true cancel, returns to `REVIEW`).
- **Signed, confirming (pending):** confirmation counter, explorer link, **no cancel** (money has moved).
- **Burn confirmed (success → advance):** node 1 flips complete, auto-advance to `TRACKING` with node 2 active.
- **Signature rejected:** `Notification` warning "You rejected the burn. Nothing was sent." → back to `REVIEW`.
- **Burn failed on-chain (mocked):** `Notification` error "The Ethereum burn failed (no funds left Ethereum)." + Retry (→ `REVIEW`) + "View on Etherscan" (mocked). This is recoverable — funds never left.
- **Insufficient gas at submit:** error "Burn couldn't be submitted — not enough ETH for gas." + Retry.

**Interactions:** auto-advance on confirm; Close → `TRACKING` (keeps running). 

**Mock data:** Ethereum tx `0xabc…123`; confirmations 0→3; ~30–90s simulated.

**Borrow from:** Send submit phase (`SubmitTransaction`) for the submitting/`Loading`/explorer-link pattern; timeline is invented.

## B6. Tracking — cross-chain pending (`TRACKING`)  ★ the headline screen

**Purpose:** Honestly represent the multi-minute, three-stage, cross-chain wait. **Resumable** — re-openable from Home/Activity and persists across wallet close. This is where the cluster earns its keep (per founder: "design the full flow"). Absorbs latency with a clear timeline + live attestation clock, never a frozen spinner.

**Layout & components**
- `SubviewHeader` title "Bridging USDC", subtitle "Ethereum → Stellar · public". Close X → Home (does NOT cancel; an in-progress chip remains on Home/Activity).
- **`BridgeTimeline`** front and center — three nodes, top→bottom:
  1. **Burning on Ethereum** — complete ✓ (with Etherscan link + "Confirmed").
  2. **Circle attesting** — active; **`AttestationClock`** "~13 min typical · 4:20 elapsed", subtext "Circle is signing off on the burn so Stellar can mint." + Circle status link (mocked). This is the longest leg — make the wait feel accounted-for.
  3. **Minting on Stellar** — pending; "Will arrive as public USDC in your Stellar account."
- **Amount + route recap** strip under the timeline: "100.00 USDC · `0x71C…3aB9` → `G…7AOO`".
- **`PublicLegBanner`** (compact): "Still public. You'll shield it after it arrives."
- **Footer:** secondary "Track in Activity" / "Close — keep tracking" (no primary action while pending). Tertiary, quiet "Get help / it's stuck" → support note.

**States (each leg has pending / active / complete / failed):**
- **Leg 2 pending (active, normal):** attestation clock counting; node pulses subtly (functional, not decorative). This is the dominant state.
- **Leg 2 — attestation slow (soft warning):** when elapsed exceeds typical (e.g. > 20 min), node turns warning-tinted: "Taking longer than usual. Circle attestations can be slow under load — your funds are safe." Still no action needed.
- **Leg 2 — attestation TIMEOUT (hard):** after a long threshold (e.g. > 60 min, mocked), `Notification` warning on the node: "Attestation hasn't completed. This is on Circle's side; your burned USDC is recoverable." Actions: **"Retry fetch attestation"** (mocked re-poll) + **"Check Circle status"** (link) + "Get help". Never claim funds are lost — CCTP is recoverable by re-claiming once attested.
- **Leg 3 — minting (active):** once attested, node 3 goes active "Minting on Stellar…" with a Stellar tx link when available.
- **Leg 3 — mint failed (recipient-not-ready, mocked):** if the Stellar account can no longer receive USDC at mint time (reserve dropped / trustline removed): `Notification` error "Couldn't mint to your Stellar account — it can't receive USDC right now. Fix that and we'll complete the mint." Actions: **"Fix and retry mint"** (re-runs invisible trustline/reserve setup, then re-attempts) + explainer. Funds remain claimable (attestation already exists) — say so.
- **All complete → arrival:** auto-advance to `ARRIVED`.
- **Resumed view (re-entered later):** identical timeline reconstructed from persisted state; if already complete while away, lands directly on `ARRIVED`/Home nudge.
- **Loading (state re-fetch on resume):** brief `Loading` "Checking bridge progress…" then render the timeline.
- **Offline / poll failure:** thin inline `Notification` warning "Can't reach the network to update progress — retrying…" with last-known timestamps; timeline shows stale-but-labeled state, never blanks out.

**Interactions:** Close → Home (chip persists); tap chip/Activity row re-enters here; explorer links open mocked pages; Retry actions re-poll. Auto-advances per leg.

**Mock data:** Leg1 done (`0xabc…123`); Leg2 attesting, 4:20 elapsed of ~13 min; Leg3 pending; amount 100.00; for timeout demo, a toggle to jump elapsed past thresholds.

**Borrow from:** No mainstream wallet reference (Freighter has no cross-chain tracker). Invent the `BridgeTimeline`. Borrow `Loading`/`Notification` scaffolding from Send, and the persistent in-progress row pattern from `views/AccountHistory` (pending-section + month-section) for the Home/Activity chip representation.

## Cross-cutting states checklist (applied across the cluster)

- **Empty:** B0 no-bridge default; B8 no active bridge.
- **Loading:** balance/route fetches (B2), connect wait (B3), re-validation (B4), state re-fetch on resume (B6).
- **Error:** bridge service down (B1), connection rejected/wrong-network/no-gas (B3), burn failed/rejected (B5), attestation timeout (B6), mint/recipient-not-ready (B2, B6), poll failure (B6).
- **Disabled:** mainnet gate everywhere; CTAs gated on preconditions (amount valid, connected, correct network, reserve present).
- **Success:** B7 arrival; B8 completed rows.
- **Pending-proof analogue:** there is **no ZK proof in the bridge itself** (bridge is public) — the pending-proof concept appears only **after** handoff, inside the Shield flow. The bridge's long-pending state is the **attestation wait** (`AttestationClock` on B6), which is the bridge's honest equivalent and must be designed with the same "never a frozen spinner" rigor.

## Honesty / anti-slop guardrails (do not violate)

- Every screen says or implies **public** at least once; never "private bridge".
- Always show **two distinct steps** (bridge in → shield); arrival is explicitly not privacy.
- No two timeline nodes look like identical cards — each names its chain + distinct state copy.
- No decorative-only motion; the only animation is the slider transition + a functional pulse on the active timeline node.
- Persistent **"to-be-validated on testnet"** ribbon + per-surface **"Mocked"** chips on: Ethereum balance, attestation timer, all explorer links, connect sheet.
- "trustline" never appears; surface only the one-time 0.5 XLM reserve when actually charged.
